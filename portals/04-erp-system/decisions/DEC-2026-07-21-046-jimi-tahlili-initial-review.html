🎯 ملخص تنفيذي (Executive Summary)

الوثائق الأربعة ممتازة كبداية — تكتب بشكل غير تقني، مع أمثلة حقيقية من DB، وشرح للعلاقات بين الموديلات. لكن فيها 3 ثغرات حرجة لازم تنعالج قبل ما تكون "مرجع للفريق":

  1. 🔴 خطير: لا يوجد ذكر لـ سياسات الـ posting_rules (template_json غامض) — المحاسب في ليبيا لازم يفهم متى يُرحّل القيد ومتى لا
  2. 🟠 ناقص مهم: لا يوجد glossary عربي موحّد (نفس المفهوم يُترجم بأكثر من طريقة: "رأس مال" / "Rapatrié"، "ترحيل" / "Posting")
  3. 🟡 ناقص: لا يوجد glossary للأحداث الـ event types — قائمة الـ 25+ نوع حدث في 03-events غير معرّفة رسمياً

تقييمي الإجمالي: 7.5/10 كوثائق تعريفية، 5/10 كمرجع تشغيلي. التوصية: إضافة 2-3 وثائق (Glossary، Posting Rules Detail، Operations Runbook) للوصول لـ 9/10.

🚨 شو خطير (What's Dangerous)

🔴 خطر #1: ادعاء "Real-time" بدون توضيح Latency

الموقع: 03-events-arabic.md يقول: "كل هذا في ثوانٍ معدودة — أوتوماتيكي بالكامل."

المشكلة: Transactional Outbox pattern هو eventual consistency — قد يتأخر ثوانٍ إلى دقائق (حسب polling interval). لا يوجد mention لـ:
- كم مرة الـ outbox processor يتفقد (SELECT ... WHERE processed_at IS NULL)
- ما هي latency المتوقعة
- ما يحدث لو processor توقّف

التأثير: مستخدم يعمل فاتورة → يتوقع المخزون ينقص "فوراً" → في الحقيقة قد يتأخر 30 ثانية. تجربة مستخدم سيئة.

التوصية: توضيح:
- Outbox Polling interval (مثلاً: 5 ثوانٍ)
- Mention أن النظام "near real-time" مش "real-time"
- Mention آلية backpressure لو processor عالق

🔴 خطر #2: posting_rules.template_json غير موثّق

الموقع: 01-finance-arabic.md يقول: "template_json | قالب القيد (JSON فيه الحسابات والشروط)"

المشكلة:
- لا يوجد schema لهذا الـ JSON
- لا يوجد مثال كامل
- "الشروط" (conditions) غامضة — ما هي الشروط المتاحة؟ كيف تُكتب؟

التأثير:
- المطور يبني JSON خاطئ → قاعدة لا تعمل أو تعمل خطأ
- المحاسب ما يقدر ينشئ قواعد جديدة (يحتاج مطور)
- لو حدث خرق — Audit trail يقول "القاعدة X تقول Y" لكن ما نعرف من كتب Y

التوصية: حرج — توثيق كامل لـ:
- template_json schema (JSON Schema رسمي)
- مثال كامل لكل event_type
- مَن يملك صلاحية تعديل القواعد (RBAC)
- مَن يملك صلاحية تفعيل/تعطيل (audit trail)

🟠 خطر #3: لا يوجد Disaster Recovery / Backup Mention

المشكلة: الوثائق لا تذكر:
- ما هي استراتيجية الـ backup
- كم مرة
- RTO / RPO
- كيف يستعيد المستخدم بياناته لو فقدت

التأثير: لو Neon عنده حادثة (نادرة لكن واردة)، المستخدم ما يعرف ما الذي سيحصل لبياناته.

التوصية: إضافة قسم قصير "النسخ الاحتياطي والاستعادة" في 04-relationships أو وثيقة منفصلة.

🟠 خطر #4: Soft Delete غير موثّق بالكامل

الموقع: 02-projects-arabic.md يذكر: deleted_at | تاريخ الحذف الناعم (soft delete) | null

المشكلة:
- ما هي استراتيجية soft delete للموديلات الأخرى؟ (هل في كل الجداول؟)
- كيف الفلترة؟ (Global query filter؟ Where clause يدوي؟)
- ما يحدث للـ FKs عند soft delete؟ (cascade؟ restrict؟)

التأثير: مطور جديد ينسى يفلتر WHERE deleted_at IS NULL → يعرض بيانات محذوفة.

التوصية: توثيق:
- قائمة الجداول التي تستخدم soft delete
- استراتيجية الفلترة (Global Filter مطلوب)
- قواعد cascade

🟡 خطر #5: Cost Center إنشاء تلقائي غير مذكور

الموقع: 02-projects-arabic.md يقول: "تنشئ مشروع → يولد cost_center تلقائياً"

المشكلة:
- ما هي خوارزمية الإنشاء (الاسم، الكود، الأب)؟
- ماذا يحدث لو فشل الإنشاء؟ (transaction rollback؟)
- هل يمكن إنشاء cost_center يدوي بدون مشروع؟

التأثير: لو عند 100 مشروع كل واحد له cost_center، ما نعرف كيف تُسمت أو كيف تُربط بحسابات CoA.

التوصية: توضيح في 02-projects أو في وثيقة منفصلة: "Cost Center Auto-Creation Rules".
