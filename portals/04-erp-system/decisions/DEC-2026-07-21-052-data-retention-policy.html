📋 PART 2: Compliance Research

2.1 IFRS / SOX (International Financial Reporting / Sarbanes-Oxley)

من PCAOB AS 1215 (Audit Documentation):

"The auditor must retain audit documentation for seven years from the date the auditor grants permission to use the auditor's report."

من Sarbanes-Oxley Section 802:

"The final rule requires that auditors retain the required documents for seven years."

ينطبق على: journal_entries, journal_lines, accounts, audit_log (when added).

2.2 GDPR (لو tenants أوروبية)

Right Data Retention Implication
Right to be forgotten Personal data (users, customers) يجب حذف عند الطلب
Right to access All personal data exportable
Data minimization Anything not needed cleanup

ينطبق على: users (email, full_name)، customers، vendors، employees (لا يوجد HR module بعد).

الـ real risk: حالياً لا يوجد tenants أوروبية (Tenants = AlFajr/Al-Burj/AlFajr Holding/الفجر، كلها ليبية). لكن multi-tenant SaaS مستقبلي قد يحتاج GDPR.

2.3 Libyan Commercial Law (قانون التجارة الليبي 1953)

من المادة (دفاتر إلزامية):

"يجب على التاجر أن يحتفظ بهذه الدفاتر والملفات لمدة خمس سنوات."

ينطبق على: journal_entries, journal_lines, accounts, الـ commercial documents.

من القانون 23 لسنة 2010 (Regarding Commercial Activities):

"These papers and documents shall be kept for five years with effect from the date of striking the company off the Commercial Register."

النتيجة: Libyan law = 5 سنوات كحد أدنى للمحاسبة.

2.4 ملخص Compliance

Standard Retention Applies to
Libyan Commercial Law 5 سنوات journal_entries, accounts
IFRS / SOX 7 سنوات نفس + audit_log
Some Libyan regulations 10 سنوات (uncertain — لازم مراجعة محاسب ليبي)
GDPR depends users (right to be forgotten)
Operational 30-90 يوم notifications, processed_events, stock_reservations

Recommended: 7 سنوات كحد أدنى (لأنه يغطي Libyan + IFRS + SOX).

🛠️ PART 4: Implementation Options

4.1 الخيار A: PostgreSQL pg_cron (مُوصى به)

Neon يدعم pg_cron (من docs):
- مُفعَّل عبر Neon API: cron.database_name parameter
- يعمل داخل الـ DB
- Standard cron syntax

مثال:

-- Enable extension
CREATE EXTENSION IF NOT EXISTS pg_cron;

-- Schedule daily cleanup of processed_events older than 30 days
SELECT cron.schedule(
  'cleanup-processed-events',
  '0 3 * * *',  -- 3 AM UTC daily
  $$DELETE FROM processed_events WHERE processed_at < NOW() - INTERVAL '30 days'$$
);

-- Schedule notifications cleanup
SELECT cron.schedule(
  'cleanup-notifications',
  '0 4 * * *',  -- 4 AM UTC daily
  $$DELETE FROM notifications 
    WHERE (is_read = true AND read_at < NOW() - INTERVAL '90 days')
       OR (is_read = false AND created_at < NOW() - INTERVAL '30 days')$$
);

Pros:
- Built-in في Neon
- No external CI/CD
- Reliable (في نفس الـ DB)
- Logs في cron.job_run_details

Cons:
- لا visual UI (SQL only)
- لا alert mechanism
- Schema changes صعبة

4.2 الخيار B: GitHub Actions Cron (Alternative)

Workflow:

# .github/workflows/cleanup.yml
name: Daily Cleanup
on:
  schedule: { cron: '0 3 * * *' }  # 3 AM UTC daily
jobs:
  cleanup:
    runs-on: ubuntu-latest
    steps:
      - name: Cleanup processed_events
        run: |
          psql "$DATABASE_URL" -c "
            DELETE FROM processed_events 
            WHERE processed_at < NOW() - INTERVAL '30 days';
          "
      - name: Cleanup notifications
        run: |
          psql "$DATABASE_URL" -c "
            DELETE FROM notifications 
            WHERE (is_read = true AND read_at < NOW() - INTERVAL '90 days')
               OR (is_read = false AND created_at < NOW() - INTERVAL '30 days');
          "
      - name: Cleanup refresh_tokens
        run: |
          psql "$DATABASE_URL" -c "
            DELETE FROM refresh_tokens 
            WHERE (expires_at < NOW() - INTERVAL '30 days')
               OR (revoked_at IS NOT NULL AND revoked_at < NOW() - INTERVAL '30 days');
          "

Pros:
- نفس pattern للـ backup (DEC-051)
- Logs في GitHub Actions
- إشعارات build failures
- Easy to modify

Cons:
- يحتاج secrets management
- GitHub Actions has limits (2000 min/month for free)
- External dependency

4.3 الخيار C: .NET BackgroundService (Hybrid)

مثال:

// Hosted/RetentionCleanupService.cs
public sealed class RetentionCleanupService : BackgroundService
{
    protected override async Task ExecuteAsync(CancellationToken ct)
    {
        while (!ct.IsCancellationRequested)
        {
            await DoCleanup(ct);
            await Task.Delay(TimeSpan.FromHours(24), ct);
        }
    }

    private async Task DoCleanup(CancellationToken ct)
    {
        // Delete processed_events > 30 days
        // Delete notifications > 90 days
        // Archive journal_entries > 7 years to R2
        // ...
    }
}

Pros:
- Type-safe
- Testable
- Reuses existing code patterns
- No external dependency

Cons:
- في نفس الـ API process (coupling)
- لا independent restart
- Hard to monitor separately

4.4 الخيار D: pg_partman (للـ partitioning)

للـ tables كبيرة جداً (millions of rows):
- journal_lines (الأكبر — كل قيد له عدة سطور)
- outbox_events (1M rows/شهر)
- audit_log (when added)

Pros:
- Auto-create partitions (by month/year)
- Auto-drop old partitions (instant)
- Query performance boost (partition pruning)

Cons:
- Schema change (migrate existing data)
- More complex to query
- Overkill لـ tables صغيرة

Phase 2: NEAR-TERM (3-5 أيام) — Compliance + Archive

Task 2.1: Archive strategy (2-3 أيام)

Workflow (GitHub Actions):

# .github/workflows/monthly-archive.yml
name: Monthly Archive
on:
  schedule: { cron: '0 5 1 * *' }  # 1st of month, 5 AM UTC
jobs:
  archive:
    runs-on: ubuntu-latest
    steps:
      - name: Archive journal_entries > 7 years
        run: |
          # Export to CSV
          psql "$DATABASE_URL" -c "
            COPY (
              SELECT * FROM journal_entries 
              WHERE entry_date < NOW() - INTERVAL '7 years'
            ) TO '/tmp/journal_entries_archive_$(date +%Y%m).csv' CSV HEADER;
          "
          # Compress
          gzip /tmp/journal_entries_archive_*.csv
          # Upload to R2
          aws s3 cp /tmp/journal_entries_archive_*.csv.gz \
            s3://erp-archive/yearly/$(date +%Y)/journal_entries_$(date +%Y%m).csv.gz
          # Delete from DB
          psql "$DATABASE_URL" -c "
            DELETE FROM journal_entries 
            WHERE entry_date < NOW() - INTERVAL '7 years';
          "

السؤال: هل كل tenant عنده cleanup منفصل؟

الإجابة: ❌ لا — cleanup job واحد لكل tenants (shared WHERE clause).

Pros:
- Simple implementation
- Predictable resource usage

Cons:
- No per-tenant customization (مثلاً: tenant كبير يبي 1y، tenant صغير يبي 7y)

الحل: لو احتاج customization:

-- Per-tenant retention (future)
DELETE FROM processed_events pe
WHERE pe.processed_at < NOW() - INTERVAL (
  SELECT COALESCE(t.custom_retention_days, 30)
  FROM tenants t
  WHERE t.id = pe.tenant_id
) * INTERVAL '1 day';
