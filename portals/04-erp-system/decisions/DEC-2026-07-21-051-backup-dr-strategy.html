🎲 PART 3: DR Scenarios (4 سيناريوهات حرجة)

السيناريو 1: Accidental Data Modification (السيناريو الأكثر احتمالاً)

الوصف: مستخدم يحدّث/يحذف بيانات بالغلط (مثلاً: DELETE FROM accounts WHERE code LIKE '5%').

التكرار المتوقع: 1-2 مرات/شهر (شائع في ERP).

الحجم: 1-100 row affected.

Recovery Strategy:
1. PITR: استعادة DB لـ "قبل الحذف" (5 دقائق)
2. Manual extraction: نسخ الـ rows المفقودة فقط من snapshot

RTO: < 1 hour
RPO: < 6 hours (Free) / < 7 days (Launch)

السيناريو الحالي: ✅ PITR كافي (Free plan) — لكن لازم المستخدم يلاحظ خلال 6 ساعات.

السيناريو 2: Bad Migration (خطر متوسط)

الوصف: migration يُطبَّق تلقائياً (AutoMigrate=true) ويكسر schema أو يحذف بيانات.

التكرار المتوقع: 1-2 مرات/ربع سنة.

الحجم: قد يكون كارثي (drop column, drop table).

Recovery Strategy:
1. Pre-migration snapshot (manual) قبل كل release
2. PITR لو الـ snapshot قديم

RTO: 1-4 hours (depends on data size)
RPO: depends on last snapshot

السيناريو الحالي: 🟠 خطر — لا manual snapshots قبل migrations! AutoMigrate=true يطبّق بدون snapshot.

التوصية: DEC-051 Phase 1 يحل هذا (Pre-migration Snapshot Playbook).

السيناريو 3: Region Failure (Neon Outage)

الوصف: Neon AWS region (aws-eu-central-1) يفشل أو عنده downtime طويل.

التكرار المتوقع: نادر (Neon SLA 99.95%).

الحجم: كارثي (لا وصول للـ DB).

Recovery Strategy:
1. External backup in different region (S3 R2 / Cloudflare R2)
2. Restore in new Neon project (region مختلف)
3. DNS switch للـ new endpoint

RTO: 4-24 hours (depends on backup size)
RPO: depends on last external backup

السيناريو الحالي: 🔴 خطر — لا external backups. كل البيانات على Neon فقط.

التوصية: DEC-051 Phase 2 يحل هذا (External pg_dump to S3/R2).

السيناريو 4: Malicious Deletion / Account Compromise (خطر أمني)

الوصف: مستخدم خبيث (أو compromised account) يحذف كل البيانات عمداً.

التكرار المتوقع: نادر لكن عالي الأثر.

الحجم: كارثي (truncate, drop).

Recovery Strategy:
1. PITR لو لاحظنا خلال 6 hours
2. External backup لو بعد 6 hours

RTO: 1-24 hours
RPO: depends on detection time

السيناريو الحالي: 🔴 خطر — يعتمد كلياً على PITR + سرعة الاكتشاف.

العلاقة بـ DEC-053: تطبيق RBAC (Admin-only for DELETE) + audit log يحدّ من هذا السيناريو. لكن لا يستبدل backup.

🎯 PART 4: RTO/RPO Targets (المقترح)

4.1 التعريفات

  • RTO (Recovery Time Objective): كم وقت للوصول لـ "running again" بعد outage؟
  • RPO (Recovery Point Objective): كم data loss مقبول (يعني: restore من متى)؟

4.2 المستهدفات (مُقترحة)

Scenario RTO Target RPO Target Mechanism
Accidental modification (small) < 1 hour < 6 hours Neon PITR (Free)
Bad migration (single table) < 4 hours Last manual snapshot Pre-migration snapshot + PITR
Region failure (full outage) < 24 hours < 24 hours External pg_dump to S3/R2 (Phase 2)
Malicious deletion (catastrophic) < 24 hours < 7 days Weekly external backup (Phase 2)

4.3 مقارنة بالخطة الحالية

Scenario Current RTO Current RPO Status
Accidental modification < 1 hour ✅ < 6 hours ✅ OK (Free plan كافي)
Bad migration < 1 hour ✅ ⚠️ depends (no snapshot) خطر
Region failure ⚠️ depends (no external) ⚠️ depends خطر كبير
Malicious deletion < 1 hour (if noticed) ⚠️ < 6 hours خطر (depends on speed)

🛡️ PART 5: Backup Strategy (3 Layers)

Layer 1: Neon PITR (Free Plan — built-in)

Always on. لا setup.

  • Window: 6 hours
  • Granularity: LSN (log sequence number)
  • Restore time: seconds
  • Cost: $0

الاستخدام: Accidental edits + bad migrations (last resort).

Layer 2: Manual Snapshots (Free Plan — 1 max)

Trigger:
- ✅ قبل كل release (sprint deploy)
- ✅ قبل كل migration خطير
- ✅ قبل أي data fix كبير

Process:

# Via Neon Console
1. Open project: lingering-feather-01780772
2. Backup & restore → Create snapshot
3. Name: "pre-sprint-3-release-2026-07-21"
4. Wait for completion (~1 minute)

# Via Neon API
POST /api/v2/projects/{project_id}/branches/{branch_id}/snapshot
{
  "name": "pre-sprint-3-release-2026-07-21"
}

القيود: 1 snapshot max في Free. لازم نحذف القديم قبل إنشاء جديد.

Layer 3: External pg_dump to S3/R2 (Phase 2 — مُقترح)

Frequency:
- ✅ Daily (عبر CI/CD)
- ✅ Retain 30 days
- ✅ Monthly (retain 1 year) — للأرشيف

Process:

# .github/workflows/backup.yml
name: Daily Backup
on:
  schedule: { cron: '0 2 * * *' }  # 2 AM UTC daily
jobs:
  backup:
    runs-on: ubuntu-latest
    steps:
      - name: pg_dump to S3
        run: |
          pg_dump "$NEON_PROD_URL" -F c -b -v \
            | gzip > backup-$(date +%Y%m%d).dump.gz
          aws s3 cp backup-*.dump.gz \
            s3://erp-backups-prod/$(date +%Y/%m/)/

التكلفة (Launch plan):
- S3 storage: ~$0.023/GB/month × 0.5 GB × 30 backups = $0.35/month
- CI minutes: ~3 min/day × 30 = 90 min/month (free for public repos)

الفايدة:
- Region-independent (لو Neon down، البيانات في S3)
- Compliance (1 year retention)
- Pre-migration snapshots (متعددة، مش 1)

Task 2.3: External pg_dump to Cloudflare R2 (2-3 أيام)

Why R2 (not S3):
- $0 egress (S3 = $0.09/GB egress)
- S3-compatible API
- Same SDK

Setup:
1. Create R2 bucket: erp-backups-prod
2. Generate R2 access key
3. GitHub Actions secret: R2_ACCESS_KEY, R2_SECRET_KEY, R2_BUCKET
4. Add GitHub Actions workflow: daily-pg-dump.yml

name: Daily pg_dump to R2
on:
  schedule: { cron: '0 3 * * *' }  # 3 AM UTC daily
jobs:
  backup:
    runs-on: ubuntu-latest
    steps:
      - name: Install postgresql client + aws-cli
        run: |
          sudo apt-get update
          sudo apt-get install -y postgresql-client
          pip install awscli

      - name: pg_dump
        env:
          DATABASE_URL: ${{ secrets.NEON_PROD_URL }}
        run: |
          pg_dump "$DATABASE_URL" -F c -b -v \
            | gzip > backup-$(date +%Y%m%d).dump.gz

      - name: Upload to R2
        env:
          R2_ACCESS_KEY: ${{ secrets.R2_ACCESS_KEY }}
          R2_SECRET_KEY: ${{ secrets.R2_SECRET_KEY }}
          AWS_ENDPOINT: ${{ secrets.R2_ENDPOINT }}
        run: |
          aws s3 cp backup-*.dump.gz \
            s3://erp-backups-prod/daily/$(date +%Y/%m/)/

      - name: Verify upload
        run: aws s3 ls s3://erp-backups-prod/daily/$(date +%Y/%m/)/

      - name: Cleanup
        run: rm -f backup-*.dump.gz

Schedule:
- Daily: 30 days retention
- Weekly: 90 days retention
- Monthly: 365 days retention
- Yearly: indefinite (للـ compliance)

Cost (R2): $0.015/GB-month storage. 0.5 GB × 30 backups = $0.22/month.

Task 2.4: Restore Test Script (1 يوم)

Deliverable: Script يختبر restore بدون تأثير على production.

#!/bin/bash
# scripts/test-restore.sh
# Usage: ./test-restore.sh <backup-file>

set -e

BACKUP_FILE=$1
TEST_DB_NAME="erp_restore_test_$(date +%s)"

# 1. Create new Neon branch (copy-on-write, instant)
echo "Creating restore branch..."
BRANCH_ID=$(neon branches create \
  --project-id "$NEON_PROJECT_ID" \
  --name "restore-test-$(date +%s)" \
  --parent "$NEON_BRANCH_ID" | jq -r '.id')

# 2. Get branch connection string
CONN_STRING=$(neon connection-string "$BRANCH_ID")

# 3. Create test database
echo "Creating test database..."
psql "$CONN_STRING" -c "CREATE DATABASE $TEST_DB_NAME"

# 4. Restore
echo "Restoring backup..."
pg_restore -d "postgresql://.../$TEST_DB_NAME?sslmode=require" "$BACKUP_FILE"

# 5. Run smoke tests
echo "Running smoke tests..."
TEST_CONN="postgresql://.../$TEST_DB_NAME?sslmode=require"
psql "$TEST_CONN" -c "SELECT COUNT(*) FROM users;"  # Should match
psql "$TEST_CONN" -c "SELECT COUNT(*) FROM accounts;"
psql "$TEST_CONN" -c "SELECT COUNT(*) FROM journal_entries;"
psql "$TEST_CONN" -c "SELECT COUNT(*) FROM projects;"

# 6. Cleanup
echo "Cleaning up..."
neon branches delete "$BRANCH_ID"

echo "Restore test PASSED ✅"

Schedule: Manual (1x per quarter, في DR drill).

🤝 PART 10: Multi-Tenant Considerations

10.1 Backup Strategy Per-Tenant

السؤال: هل كل tenant عنده backup منفصل؟

الإجابة: ❌ لا — كل tenants في نفس DB، فالـ backup = DB كامل.

الاستعادة Per-Tenant (لو tenant واحد فقط تالف):

  1. Restore DB كامل على branch جديد
  2. Export rows للـ tenant المتضرر:
    sql -- Extract one tenant pg_dump --table='accounts' --where='tenant_id = X' ...
  3. Import للـ rows في production

التعقيد: Medium (manual SQL work).

10.2 Per-Tenant Snapshot (مستقبلي)

خيار: إنشاء Neon branch per-tenant (10 branches = $0.20/month extra).
- كل branch = snapshot للـ tenant data فقط
- PITR per-tenant

Trade-off: $0.20/month × 4 tenants = $0.80/month. يستحق؟ ❌ حالياً (4 tenants فقط).

10.3 Tenant Data Sensitivity

Tenant Data Sensitivity Backup Priority
AlFajr Trading عالي (real data) 🟠 High
Al-Burj Construction عالي (real data) 🟠 High
AlFajr Holding عالي (consolidated) 🔴 Critical
الفجر (جديد) منخفض (test) 🟢 Low

Note: كل tenants في نفس الـ backup (لا يوجد segregation). RTO/RPO = نفسه للكل.
