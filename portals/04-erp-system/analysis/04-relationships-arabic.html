🌍 الصورة الكبيرة (Big Picture)

النظام فيه 12 موديل كل واحد مسؤول عن شغلة معينة. لكنهم مش جزر منعزلة — مرتبطين بـ:

  1. 🔗 Foreign Keys (روابط مباشرة في الـ DB)
  2. 📬 Events (رسايل بين الموديلات)
  3. 💰 Cost Centers (محاور مشتركة بين Projects و Finance)
  4. 🏢 Tenant (الجذر المشترك لكل البيانات)

🏛️ الجذر: Tenants (المستأجرين)

كل البيانات في النظام مرتبطة بـ tenant_id
= كل شركة (أو مجموعة شركات) عندها data خاصة فيها
= Multi-tenant architecture

Tenant 1: AlFajr Trading & Contracting
Tenant 2: Al-Burj Construction & Contracting
Tenant 3: AlFajr Holding
Tenant 4: الفجر (جديد)

كل جدول عنده tenant_id (إلا القليل مثل VersionInfo).

🪪 Identity (الهوية) — الجذر التقني

Tenants
  └── Users
        └── Roles (عبر user_roles)
              └── Permissions (داخل الـ role)

Refresh Tokens (للجلسات)
Password Reset Tokens (لإعادة تعيين كلمة المرور)
Audit Log (مين عمل شنو)

Authentication flow:

1. User يكتب email + password
2. System يبحث في users table
3. يتحقق من password_hash (bcrypt)
4. يطلع JWT token
5. يحفظ refresh token في refresh_tokens
6. يحدّث last_login في audit_log

🏢 Companies (الشركات)

Tenants (الشركة القابضة)
  └── Companies (الشركات الفرعية)
        ├── Users (ممكن يكون مرتبط بشركة محددة)
        ├── Accounts (دليل حسابات)
        ├── Journal Entries
        ├── Vendors
        ├── Customers
        ├── Items
        ├── Employees
        └── Projects

Multi-Company = داخل نفس الـ tenant ممكن يكون عدة شركات (subsidiaries).
كل عملية ممكن تنسب لشركة محددة.

💰 Finance ↔ كل الموديل الثانية

Finance ↔ Procurement (المشتريات):

عند استلام فاتورة من مورد (vendor_bill):
  vendor_bill → يحفظ في accounts payable
  posting_rule → يطلع journal_entry:
    - مدين: حساب مصروفات (5100)
    - دائن: حساب الدائنين (2100)

عند الدفع:
  payment → ينقص من account payable
  posting_rule → يطلع journal_entry:
    - مدين: حساب الدائنين (2100)
    - دائن: البنك (1202)

Finance ↔ Sales (المبيعات):

عند إصدار فاتورة (sales_invoice):
  posting_rule → يطلع journal_entry:
    - مدين: المدينين (1203) — بالمبلغ
    - دائن: إيرادات (4100)
    - دائن: ضريبة مخرجات (لو موجودة)

عند استلام دفعة:
  payment → ينقص من المدينين
  posting_rule → يطلع journal_entry:
    - مدين: البنك (1202)
    - دائن: المدينين (1203)

Finance ↔ HR (الرواتب):

Payroll run → يحسب صافي الرواتب
  posting_rule → يطلع journal_entries متعددة:
    - مدين: مصروف رواتب (5001) — إجمالي
    - دائن: ضريبة رواتب مستحقة (2100) — لو في
    - دائن: خصومات (2100)
    - دائن: البنك (1202) — صافي

Finance ↔ Inventory (المخزون):

عند استلام بضاعة (GRN):
  posting_rule → يطلع journal_entry:
    - مدين: المخزون (1300) — قيمة البضاعة
    - دائن: حساب مؤقت (2110) — awaiting vendor bill

عند إصدار vendor bill:
  posting_rule → يطلع journal_entry:
    - مدين: حساب مؤقت (2110)
    - دائن: الدائنين (2100) — المبلغ المستحق

عند بيع بضاعة:
  posting_rule → يطلع journal_entry:
    - مدين: تكلفة البضاعة المباعة (5100)
    - دائن: المخزون (1300)

Finance ↔ Projects (المشاريع):

كل مشروع → cost_center_id
كل مصروف → journal_line.cost_center_id
التقرير: تكلفة المشروع = sum(القيود) where cost_center = project.cost_center_id

مثال:
  project_budgets:
    - project_id: P-2026-001
    - cost_center_id: CC-001
    - account_id: (حساب الميزانية)
    - budget_amount: 450,000

  لما يصير مصروف:
    - journal_line.cost_center_id = CC-001
    - spent_amount يتحدث

هذا أقوى ربط — لأن كل مصروف ممكن ينسب لمشروع.

📁 Projects ↔ HR + Procurement + Inventory

Projects ↔ HR (الموارد البشرية):

المشروع يحتاج عمالة:
  resource_assignments:
    - project_id: P-2026-001
    - task_id: T-001
    - resource_id: R-002 (عامل بناء)
    - user_id: U-anas (الموظف)
    - from: 2026-01-15
    - to: 2026-04-30
    - hourly_rate: 25 LYD

التكلفة = actual_hours × hourly_rate
   = 200 hours × 25 = 5,000 LYD

Projects ↔ Procurement (المشتريات):

المشروع يحتاج مواد:
  1. Project → PO (purchase_order.project_id)
  2. PO → items (purchase_order_lines)
  3. عند الاستلام → GRN (goods_receipts.project_id)
  4. الفاتورة → vendor_bill → قيد على cost_center

Projects ↔ Inventory (المخزون):

عند صرف مواد للمشروع:
  - stock_movement: 
    - item_id: I-cement
    - quantity: -100 (خرج)
    - project_id: P-2026-001
    - cost_center_id: CC-001
  - posting_rule: قيد تكلفة

النتيجة: المخزون ينقص + المصروف يُسجّل على المشروع

📬 Events (الأحداث) — العمود الفقري

[موديل A] → outbox_events → [مستمعين Events] → [موديلات B, C, D]

أمثلة من Finance و Projects:

Finance event: "journal_entry.posted"
  مستمعين:
    - Reports: يحدث الأرقام
    - Notifications: يبعت للمحاسب
    - Audit: يسجل في audit_log
    - Budget checker: يتحقق من الميزانية

Projects event: "project_budget_exceeded"
  مستمعين:
    - Notifications: يبعت للمدير (alert)
    - Audit: يسجل التجاوز
    - Finance: يضيف ضريبة/رسوم إضافية

Projects event: "project_task.completed"
  مستمعين:
    - Project: يحدث progress_percent
    - Reports: يحدث KPIs
    - HR: يحسب ساعات العمل الفعلية
    - Notifications: يبعت للعميل (لو project لعميل)

🎯 سيناريو كامل: "مشروع من البداية للنهاية"

1️⃣ Setup (الشروع):
   - Project: P-2026-001 (مشروع بناء فيلات)
   - cost_center: CC-001 (auto-created)
   - budget: 450,000 LYD

2️⃣ Planning (التخطيط):
   - Tasks: 10 tasks (الأساسات، الهيكل، التشطيبات...)
   - Resources: عمال + معدات
   - Assignments: مين يعمل شنو ومتى

3️⃣ Execution (التنفيذ):
   - كل يوم:
     - العامل يدخل ساعات → resource_assignments
     - شراء مواد → PO → GRN → stock_movement
     - مصاريف → journal_entry → cost_center=CC-001
   - project_budgets.spent_amount يتحدث أوتوماتيك

4️⃣ Monitoring (المراقبة):
   - 80% spent → notification "تحذير"
   - 100% spent → notification "تجاوز" + audit
   - progress_percent < 50% عند 80% time → "behind schedule"

5️⃣ Closing (الإغلاق):
   - project.status = 3 (Closed)
   - حساب الفعلي vs المخطط
   - فاتورة العميل (لو في) → journal_entry إيراد
   - تقرير ربحية المشروع

كل هذا مترابط — ما تحتاج تدخل يدوي في كل خطوة.

🛡️ Multi-Tenancy + Security

كل query في الـ API يفلتر بـ tenant_id
= Tenant A ما يشوف بيانات Tenant B
= حتى لو في نفس الـ DB

الـ Security:
  - JWT token فيه tenant_id
  - كل request يتحقق من token
  - كل query يضيف WHERE tenant_id = ?
  - لا يمكن للـ user يدخل tenant غير بتاعه

هذي أقوى حماية — لا يمكن تسرب بيانات بين الشركات.
