🎯 شو هو Event Catalog؟

هو المرجع الرسمي لكل نوع حدث في النظام: شو هو، مَن ينتجه، مَن يستهلكه، شكل الـ payload، شو يأثر في DB.

بدونه: مطوّر جديد يضيف event → يخترع naming convention خاص → يتعارض مع events ثانية.

معه:
- إضافة event جديدة = إضافة صف للجدول + JSON Schema + handler
- تنسيق بين الموديلات (Finance, Inventory, Projects, ...)
- Audit trail واضح لكل integration

🏗️ Outbox Schema (من outbox_events)

من OutboxEvent.cs + OutboxRepository.cs:

العمود النوع المعنى ملاحظات
id UUID (PK) معرّف الصف generated
tenant_id UUID المستأجر للـ routing + isolation
event_type VARCHAR اسم الـ C# record (مثل "StockReceivedEvent") من typeof(T).Name
aggregate_id UUID EventId (dedup key) ⚠️ ليس ID الكيان الفعلي — للتوضيح
aggregate_type VARCHAR نوع الكيان الجذري ("StockMovement", "JournalEntry", ...) من switch في AggregateTypeOf()
payload TEXT (JSON) محتوى الحدث (camelCase) يُسلسل عبر JsonSerializer.Serialize
occurred_at TIMESTAMP متى وقع الحدث من @event.OccurredAt
processed_at TIMESTAMP? متى اتعالج (NULL = pending) يُحدّث عبر MarkProcessedAsync
retry_count INT عدد المحاولات الفاشلة يبدأ من 0
max_retries INT الحد الأقصى Default: 3 (من EventBus)
last_error VARCHAR(4000)? رسالة آخر خطأ يُقتطع لـ 4000 حرف

Index: لا يوجد حالياً index صريح! (Production-grade TODO)

⚠️ مشكلة أداء: FetchUnprocessedAsync يستخدم WHERE processed_at IS NULL AND retry_count < max_retries ORDER BY occurred_at LIMIT 50 — بدون index، هذا full table scan عند نمو البيانات.

التوصية: 

CREATE INDEX ix_outbox_pending 
ON outbox_events (occurred_at) 
WHERE processed_at IS NULL AND retry_count < max_retries;

📜 Integration Event Contract (من IIntegrationEvent)

public interface IIntegrationEvent
{
    Guid EventId { get; }      // Dedup key (idempotency)
    Guid TenantId { get; }     // Routing (أي tenant)
    DateTime OccurredAt { get; }  // Ordering + audit
}

كل event record يجب أن يطبّق هذا الـ interface.

مثال StockReceivedEvent:

public record StockReceivedEvent(
    Guid EventId, 
    Guid TenantId, 
    Guid StockMovementId,    // ← ID الكيان الفعلي
    Guid ItemId, 
    Guid WarehouseId, 
    decimal Quantity, 
    decimal UnitCost,
    string? PurchaseOrderRef, 
    DateTime OccurredAt
) : IIntegrationEvent;

📋 الـ 5 Events بالتفصيل

1️⃣ StockReceivedEvent ✅ Active

Purpose: استلام بضاعة من مورد أو تحويل (+). يحدث قيد مالي + تحديث تكلفة المخزون.

Producer:
- StockMovementService.PostAsync(movementType: "Receive")
- في نفس الـ transaction: يحدث stock_level + يطلق event

Consumer:
- StockReceivedEventHandler (من EventHandlers.cs)
- يستدعي PostingRulesService.ApplyRulesAsync(StockReceived, payload)

Effect على DB:
1. stock_movements row (status=Posted)
2. stock_levels UPSERT (item_id, warehouse_id, qty += quantity)
3. outbox_events row (event_type='StockReceivedEvent')
4. (بعد 5s) journal_entries row + journal_lines rows
5. (بعد 5s) outbox_events.processed_at يُحدّث
6. (بعد 5s) processed_events row

JSON Schema (Payload):

{
  "eventId": "uuid",
  "tenantId": "uuid",
  "stockMovementId": "uuid",
  "itemId": "uuid",
  "warehouseId": "uuid",
  "quantity": 100,
  "unitCost": 50.00,
  "purchaseOrderRef": "PO-2026-0042",
  "occurredAt": "2026-07-21T20:30:00Z"
}

القيود:
- quantity > 0 (إجباري)
- unitCost ≥ 0
- purchaseOrderRef اختياري
- حساب المبلغ = quantity × unitCost (يُمرَّر كـ payload.Amount)

3️⃣ StockTransferredEvent 🟡 Emitted but no handler

2. stock_levels من المخزن المصدر -= qty
3. stock_levels للمخزن الهدف += qty
4. outbox_events row
5. (بعد 5s) ❌ لا handler → "Event has unknown event type" warning

JSON Schema (Payload):

  "tenantId": "uuid",
  "stockMovementId": "uuid",
  "itemId": "uuid",
  "fromWarehouseId": "uuid",

  "quantity": 50,
  "unitCost": 50.00,
  "occurredAt": "2026-07-21T20:30:00Z"

⚠️ Note: هذا الحدث يُسجَّل لكن لا يُعالج. قد ينمو outbox_events بدون فائدة. التوصية: إما إضافة handler (analytics) أو عدم إطلاقه أصلاً.

🔄 Lifecycle (7 Stages)

[Stage 1: Business Operation]
   ↓
   User: POST /api/inventory/movements/{id}/post
   ↓
   StockMovementService.PostAsync()
   ├─ Update movement (status=Posted)
   ├─ UPSERT stock_level
   ├─ INSERT outbox_event  ← Stage 2 starts here
   └─ COMMIT (atomic with Stage 1)

[Stage 2: Event Persisted]
   ↓
   outbox_events row exists (processed_at = NULL)

[Stage 3: Response Sent]
   ↓
   HTTP 200 OK to user (fast, no waiting)

[Stage 4: Background Polling]
   ↓
   OutboxProcessorHostedService (every 5s)
   FetchUnprocessedAsync(50)
   ↓
   [List of up to 50 unprocessed events]

[Stage 5: Idempotency Check]
   ↓
   For each event:
     IsProcessedAsync(eventId)?
     ├─ YES → MarkProcessedAsync (skip) → continue
     └─ NO ↓

[Stage 6: Dispatch]
   ↓
   Deserialize payload → find handler via DI
   ↓
   IIntegrationEventHandler<T>.HandleAsync(evt, ct)
   ├─ Calls PostingRulesService.ApplyRulesAsync
   ├─ Creates JournalEntry
   └─ Posts JournalEntry

[Stage 7: Mark]
   ↓
   ├─ MarkProcessedAsync (processed_events row inserted)
   ├─ outbox_events.processed_at = NOW
   └─ Log: "Event X processed"

[Failure Path]
   ↓
   if (exception in Stage 6):
     retry_count++
     last_error = ex.Message (truncated 4000)
     if (retry_count >= max_retries):
       MarkProcessedAsync (gives up) + ERROR log
       Admin must use POST /api/events/retry/{id}

زمن end-to-end: في أفضل حالة 5 ثوان (polling interval). في أسوأ حالة (handler بطيء) = ~10-15s. ليس real-time.

🌐 API Endpoints (من AGENTS.md)

Method Path الوصف Auth
GET /api/events/outbox قائمة events معلقة (pending) للـ tenant ✅ JWT
GET /api/events/processed قائمة events معالَجة (audit trail) ✅ JWT
GET /api/events/pending-count عدد events معلقة فقط (headcount) ✅ JWT
POST /api/events/retry/{id} إعادة محاولة يدوية (admin) — retry_count=0, last_error=null ✅ JWT + Admin

Query params لـ GET:
- skip (default 0) + take (default 50)
- unprocessedOnly (bool, default false)

مثال:

curl https://api.erp.com/api/events/outbox?unprocessedOnly=true&skip=0&take=20

🎯 مثال End-to-End كامل: StockReceivedEvent

-- T=0: User posts GRN via UI
POST /api/inventory/movements/M-001/post
{
  "movementType": "Receive",
  "itemId": "I-cement",
  "warehouseId": "W-main",
  "quantity": 100,
  "unitCost": 50.00,
  "purchaseOrderRef": "PO-2026-0042"
}

-- T=0.05s: Service in transaction
BEGIN TRANSACTION
  UPDATE stock_movements SET status='Posted' WHERE id='M-001'
  UPSERT stock_levels SET qty = qty + 100 WHERE item='I-cement' AND warehouse='W-main'
  INSERT INTO outbox_events (id, event_type='StockReceivedEvent', aggregate_id=event.EventId, payload='{...}', occurred_at=NOW())
COMMIT

-- T=0.06s: HTTP 200 to user
{"status":"Posted"}

-- T=5s: OutboxProcessor polls
SELECT * FROM outbox_events WHERE processed_at IS NULL ORDER BY occurred_at LIMIT 50
-- Picks our event

-- T=5.01s: Idempotency check
SELECT 1 FROM processed_events WHERE event_id = '...' 
-- Result: NULL → not processed yet → proceed

-- T=5.02s: Dispatch
Deserialize payload → StockReceivedEvent record
DI lookup: IIntegrationEventHandler<StockReceivedEvent> → StockReceivedEventHandler
Call: StockReceivedEventHandler.HandleAsync(evt, ct)
  └─ IPostingRulesService.ApplyRulesAsync(tenantId, userId, StockReceived, payload)
       └─ Find active rules for StockReceived
       └─ Deserialize template_json
       └─ For each line:
            └─ AccountCode 1300 → resolve to AccountId
            └─ AmountFormula "{amount}" → payload.Quantity * payload.UnitCost = 5000
            └─ Add to journal lines: Dr 1300 = 5000
            └─ For credit line: AccountCode 2100 → resolve
            └─ Add: Cr 2100 = 5000
       └─ Validate: totalDebit (5000) == totalCredit (5000) ✓
       └─ CreateDraftAsync(JournalEntry)
       └─ PostAsync(JournalEntry) → Dr Inventory 5000, Cr A/P 5000

-- T=5.5s: Mark processed
INSERT INTO processed_events (event_id, tenant_id, processed_at) VALUES (...)
UPDATE outbox_events SET processed_at = NOW() WHERE id = '...'

-- T=5.51s: Log
[INFO] Event 'StockReceivedEvent' (id=...) processed — JournalEntry JE-2026-0042 created

-- User now sees the journal entry in UI

Total time: ~5.5s من POST لـ JournalEntry visible.

🛡️ Governance: مَن يضيف event جديد؟

Process (3 خطوات):

  1. DEC رسمي:
  2. افتح DEC-XXX (Event Addition)
  3. حدّد: اسم الـ event، JSON Schema، Producer، Consumer، Effect

  4. أضف صف لـ Master Table هنا (Catalog)

  5. Implementation:

  6. أضف record XxxEvent(...) : IIntegrationEvent
  7. أضف IIntegrationEventHandler<XxxEvent> + class
  8. DI Register: services.AddScoped<IIntegrationEventHandler<XxxEvent>, XxxEventHandler>()
  9. أضف case في OutboxProcessorHostedService.DispatchAsync (switch)

  10. Publish في business logic: await _eventBus.PublishAsync(new XxxEvent(...), ct)

  11. Tests:

  12. Test للـ handler (successful path)
  13. Test لـ retry logic (handler throws)
  14. Test لـ idempotency (same EventId twice)

⚠️ ممنوع:

  • ❌ إضافة event بدون JSON Schema موثّق
  • ❌ تخطي الـ IIntegrationEvent interface (EventId/TenantId/OccurredAt إجبارية)
  • ❌ إطلاق event من خارج transaction (يكسر atomicity guarantee)
  • ❌ تعديل event existing بدون versioning (سيكسر consumers)

🔢 Versioning Strategy

عند تغيير payload لحدث موجود:

  1. Backward-compatible changes (إضافة field اختياري):
  2. ✅ يمكن بدون version جديد

  3. Consumers الحاليون يتجاهلون الـ field الجديد

  4. Breaking changes (تغيير/حذف field، تغيير type):

  5. ❌ لا تعدّل الـ event الموجود
  6. ✅ أنشئ event جديد: XxxEventV2
  7. أطلق الاثنين لفترة انتقالية

  8. أوقف V1 بعد ما كل consumers يهاجر

  9. Schema في Master Table:

  10. version column (V1, V2, ...)
  11. deprecated_at (timestamp)
  12. replaced_by (V2)

⚠️ Production-Grade TODOs (من الكود)

من OutboxProcessorHostedService.cs comments + تحليل:

1. Polling → LISTEN/NOTIFY (موجود في TODO)

الحالي: 

private static readonly TimeSpan PollInterval = TimeSpan.FromSeconds(5);

المستقبلي: Postgres LISTEN/NOTIFY (push-based) — latency → ~10ms.

التأثير: حالياً latency = 5s ثابت. في multi-tenant ثقيل، قد يكون الـ queue طويلاً.

2. Multi-instance Safety (موجود في TODO)

الحالي: Single-instance فقط. لو شغّلت 2 instances من الـ backend:
- كلاهما يقرأ نفس الـ outbox events
- كلاهما يحاول يطبق القواعد
- النتيجة: قيود مكررة + race conditions

المستقبلي: FOR UPDATE SKIP LOCKED على outbox_events:

SELECT ... FROM outbox_events 
WHERE processed_at IS NULL
ORDER BY occurred_at
LIMIT @Batch
FOR UPDATE SKIP LOCKED;

3. processed_events Expiry (مذكور في AGENTS.md)

الحالي: processed_events ينمو forever.

المستقبلي: TTL (مثلاً 30 يوم) + cleanup job.

التأثير: عند 1M events/يوم = 30M rows بعد شهر. Storage cost.

4. Index على outbox_events (مكتشف في هذا الـ Catalog)

المشكلة: FetchUnprocessedAsync بدون index → full table scan.

الإصلاح: 

CREATE INDEX ix_outbox_pending 
ON outbox_events (occurred_at) 
WHERE processed_at IS NULL AND retry_count < max_retries;
