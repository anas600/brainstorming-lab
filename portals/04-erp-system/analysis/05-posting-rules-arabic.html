🎯 شو هو Posting Rules Engine؟

هو الدماغ اللي يربط الأحداث بقيود اليومية. كل مرة يصير حدث في النظام (استلام بضاعة، إصدار فاتورة، استلام دفعة...)، المحرّك يفحص: "في قاعدة تنطبق على هذا الحدث؟" → لو نعم → يُنشئ قيد يومية تلقائياً.

مثال واقعي:

1. مستخدم يستلم 100 كيس اسمنت (قيمة 5,000 LYD)
2. النظام ينشئ stock_movement + يطلق event: StockReceived
3. Outbox Processor يلتقط الحدث (كل 5 ثوان)
4. PostingRulesService يبحث: "في قاعدة StockReceived مفعلة؟"
5. يجد القاعدة الافتراضية: "1300 → debit, 2100 → credit, {amount}"
6. ينشئ journal_entry + lines
7. يُرحّل القيد تلقائياً (Draft → Posted)
8. ينشر event جديد: JournalEntryPosted (للمستهلكين الآخرين)

النتيجة: في ثوانٍ، قيد محاسبي مُرحّل في الدفاتر ✅

بدون المحرّك: المحاسب يكتب كل قيد يدوياً. مع 100+ عملية/يوم = بطيء + أخطاء.

🏗️ بنية القاعدة (PostingRule Schema)

الجدول في DB (posting_rules):

العمود النوع المعنى ملاحظات
id UUID (PK) معرّف فريد generated
tenant_id UUID (FK) المستأجر (multi-tenant) إجباري
name VARCHAR(200) اسم وصفي للقاعدة مثلاً: "استلام بضاعة"
description VARCHAR(500) وصف تفصيلي اختياري
event_type INT نوع الحدث (enum) قيم: 1-4 (انظر أدناه)
is_active BOOL هل القاعدة مفعّلة؟ Default: true
template_json TEXT قالب القيد (JSON) الإجباري الأهم
created_at TIMESTAMP تاريخ الإنشاء
updated_at TIMESTAMP آخر تحديث

Index: ix_posting_rules_tenant_event على (tenant_id, event_type, is_active) — للبحث السريع عن القواعد النشطة لحدث معيّن.

📋 أنواع الأحداث المدعومة (TriggeringEvent enum)

الكود يعرّف 4 أنواع فقط حالياً (Phase 1):

public enum TriggeringEvent
{
    StockReceived = 1,      // استلام بضاعة (من مورد أو تحويل)
    StockIssued = 2,        // صرف بضاعة (للبيع أو مشروع)

Event متى يطلق؟ الحالة

StockIssued عند تسجيل صرف بضاعة ✅ مفعّل (مع قاعدة افتراضية)
InvoiceCreated عند إنشاء فاتورة مبيعات/مشتريات 🟡 Placeholder (لا handler بعد)
PaymentReceived عند استلام دفعة 🟡 Placeholder (لا handler بعد)

📐 بنية Template (PostingRuleTemplate)

هذا هو الجزء الأهم — قالب القيد بصيغة JSON. مصمّم كـ POCO (Plain Old CLR Object) ليسهل deserialization.

POCO Classes (من PostingRule.cs):

public class PostingRuleTemplate
{
    public string Description { get; set; } = string.Empty;
    public string? Reference { get; set; }
    public List<PostingRuleLineTemplate> Lines { get; set; } = new();
}

public class PostingRuleLineTemplate
{
    public string AccountCode { get; set; } = string.Empty;  // "1300" (يحلّ لـ ID وقت التشغيل)
    public string Side { get; set; } = "debit";              // "debit" أو "credit"
    public string AmountFormula { get; set; } = "0";         // صيغة المبلغ
}

مثال كامل (JSON Schema):

{
  "Description": "استلام بضاعة من مورد",
  "Reference": null,
  "Lines": [
    {
      "AccountCode": "1300",
      "Side": "debit",
      "AmountFormula": "{amount}"
    },
    {
      "AccountCode": "2100",
      "Side": "credit",
      "AmountFormula": "{amount}"
    }
  ]
}

القواعد الإجبارية للتحقق (من PostingRulesService.CreateAsync):

  1. ✅ كل AccountCode يجب أن يكون موجود في CoA (دليل الحسابات) للـ tenant
  2. ✅ كل حساب يجب أن يكون IsPostable = true (مش حساب تجميعي)
  3. ✅ عدد السطور ≥ 2 (لا قاعدة من سطر واحد)
  4. ✅ Σ(debit) = Σ(credit) — وإلا يتخطى النظام القاعدة ويكتب error في الـ log

🧮 صيغ المبالغ المدعومة (AmountFormula)

من PostingRulesService.EvaluateFormula — الدالة بسيطة جداً:

private decimal EvaluateFormula(string formula, EventPayload payload)
{
    if (string.IsNullOrWhiteSpace(formula)) return 0;
    var trimmed = formula.Trim();

    if (trimmed == "{amount}") return payload.Amount;          // المبلغ الكامل
    if (trimmed == "{amount}*2") return payload.Amount * 2;    // الضعف (حالة خاصة)
    if (decimal.TryParse(trimmed, out var n)) return n;        // رقم خام

    _logger.LogWarning("صيغة مبلغ غير معروفة: {Formula}", formula);
    return 0;  // ⚠️ صامت — لا يرمي exception، فقط log
}

الصيغ المدعومة:

الصيغة المعنى مثال
{amount} المبلغ الكامل من الـ payload 5,000.00
{amount}*2 ضعف المبلغ (حالة خاصة فقط) 10,000.00
123.45 رقم خام (decimal literal) 123.45
0 صفر (لحالات نادرة) 0
أي صيغة أخرى 0 + log warning ⚠️ —

⚠️ قيود خطيرة (خلل في التصميم؟):

  1. لا يوجد تقسيم: {amount}/2 مش مدعوم
  2. لا يوجد جمع: {amount}+100 مش مدعوم
  3. لا يوجد شرط/حسابات: لا if/case
  4. fallback صامت: لو الصيغة غلط → المبلغ = 0 → القيد قد يكون غير متوازن → النظام يتخطاه بصمت

التوصية: توسيع EvaluateFormula لدعم:
- العمليات الحسابية الأساسية (+, -, *, /)
- متغيرات متعددة ({amount}, {tax}, {quantity})
- صيغة percentage (مثلاً: {amount}*0.15 للضريبة 15%)

🔄 Flow كامل: من حدث إلى قيد مُرحّل

[1] مصدر الحدث (مثلاً: Inventory Service)
   ↓
   ينشئ stock_movement row
   ينشر event في outbox_events (نفس الـ transaction)
   ↓
[2] OutboxProcessorHostedService (Background Service)
   ↓
   كل 5 ثوان: يجيب events غير معالجة (batch = 50)
   يفحص idempotency (processed_events)
   ↓
[3] DispatchAsync (في OutboxProcessor)
   ↓
   يفك event payload (Deserialize)
   يجد الـ IIntegrationEventHandler المناسب
   ↓
[4] EventHandler (مثلاً: StockReceivedEventHandler)
   ↓
   يستدعي IPostingRulesService.ApplyRulesAsync(
       tenantId, userId, StockReceived, payload)
   ↓
[5] PostingRulesService.ApplyRulesAsync
   ↓
   يجيب القواعد النشطة: ListActiveByEventAsync
   لكل قاعدة:
     - Deserialize TemplateJson
     - لكل سطر: GetByCodeAsync(AccountCode)
     - EvaluateFormula لكل AmountFormula
     - يحسب Σ(debit) و Σ(credit)
     - لو غير متوازن → skip + log error
     - ينشئ journal_entry (Draft)
     - يرحّله (Post)
   ↓
[6] ينشر event جديد: JournalEntryPosted
   ↓
   في outbox_events → يدخل cycle جديدة (للمستهلكين: Notifications, Reports, ...)

زمن التنفيذ: في أفضل حالة 5 ثوان (Polling interval). في أسوأ حالة قد يتأخر لو backlog كبير.

🌱 القواعد الافتراضية (EnsureDefaultRulesAsync)

عند إنشاء tenant جديد، النظام يضيف قاعدة افتراضية لكل event type تلقائياً.

مثال: StockReceived (من PostingRulesService.EnsureDefaultRulesAsync):

TemplateJson = JsonSerializer.Serialize(new PostingRuleTemplate
{

القيد المحاسبي:
| السطر | الحساب | الكود | الجانب | المبلغ |

🛡️ مَن يستطيع إنشاء/تعديل القواعد؟ (RBAC)

⚠️ نقطة ضعف في التوثيق الحالي:

من PostingRulesController:

[ApiController]

[Authorize]   // ← فقط JWT، لا role-based

ما نعرفه:
- ✅ لازم يكون مسجّل دخول (JWT)
- ❓ لا يوجد [Authorize(Roles = "Admin")] صريح
- ❓ لا يوجد تمييز بين محاسب ومطوّر ومستخدم عادي

الافتراض (لم يُتحقق بعد):
- حسب نظام RBAC العام، المفترض Admin أو Accountant فقط
- لكن هذا غير موثّق في الكود

التوصية: فتح DEC منفصل لتوثيق RBAC الكامل (DEC-048 مقترح).

⚠️ قيود معروفة (Limitations)

1. لا يوجد Validation للـ User Input قبل الترحيل

في ApplyRulesAsync:

// معادلة الـ double-entry
if (totalDebit != totalCredit)
{
    _logger.LogError("قاعدة {RuleId} تنتج قيد غير متوازن: D={D} C={C} — تم التخطي",
        rule.Id, totalDebit, totalCredit);
    continue;  // ← يتخطى بصمت
}

المشكلة: لو القاعدة تنتج قيد غير متوازن، ما يرمي exception → فقط log error. النتيجة: الحدث يضيع، والقيد ما يُنشأ، والمحاسب ما يعرف.

التوصية: إضافة alert عبر notifications table: "فشل تطبيق القاعدة X للقيد Y — القيد غير متوازن".

2. EventPayload.currency لا يُستخدم في القيد

الـ template_json يستخدم AccountCode فقط. لا يوجد ربط بين العملة والحساب. لو المبلغ بالـ USD وحساب 1300 بالـ LYD → مشكلة محاسبية (مخاطرة سعر صرف).

التوصية: إضافة currency إلى journal_lines + تحقق تطابق العملة في القاعدة.

3. لا يوجد Audit Trail لتعديلات القواعد

PostingRuleRepository.UpdateAsync موجود لكن لا يُسجّل في audit_log من رأى التعديل. مخالف لتوصيات SOC2/IFRS.

4. لا يوجد Dry-Run Mode

لو محاسب يبني قاعدة معقدة، ما يقدر "يجرّب" بدون تطبيق فعلي. الـ trigger/{eventType} يطبق فعلاً (ينشئ قيد).

التوصية: إضافة ?dryRun=true query param.

5. تكرار القواعد لنفس EventType ممكن

في ApplyRulesAsync، كل القواعد النشطة تُطبّق. لو في قاعدتين لـ StockReceived → قيدان يومياً لكل استلام. هذا قد يكون مقصود (مثلاً: عادي + ضريبة) لكن خطير لو بالغلط.

التوصية: تحذير في UI لو في >1 قاعدة نشطة لنفس event.
