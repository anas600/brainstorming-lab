🎯 شو هو RBAC في ERP-SYSTEM؟

RBAC = Role-Based Access Control — نموذج أمني يربط:
- 👤 مستخدم (User) — شخص يستخدم النظام
- 🎭 دور (Role) — "محاسب"، "مدير"، "مشاهد"
- 🛡️ إذن (Permission) — حق الوصول لعملية معيّنة (يُستنتج من Role)

في ERP-SYSTEM:

User ─many-to-many─> Role
                       │
                       └──> Permissions (مُعرّفة ضمنياً عبر Role Name)

ملاحظة: النظام لا يخزّن Permissions كجدول منفصل — الصلاحيات مُضمّنة في اسم الـ Role (Convention over Configuration).

🏗️ البنية الحالية (Current Implementation)

1️⃣ جدول roles (من 20260614_120000_CreateIdentityTables.cs)

CREATE TABLE roles (
  id UUID PRIMARY KEY,
  tenant_id UUID NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
  name VARCHAR(100) NOT NULL,
  description VARCHAR(500) NOT NULL DEFAULT '',
  created_at TIMESTAMP NOT NULL,
  UNIQUE INDEX ix_roles_tenant_name (tenant_id, name)
);

ملاحظات:
- ✅ tenant_id — كل دور مربوط بمستأجر (multi-tenant)
- ✅ UNIQUE (tenant_id, name) — لا يمكن إنشاء دور مكرر داخل نفس الـ tenant
- ❌ لا يوجد is_system — لا يمكن التمييز بين "دور نظام" (لا يُحذف) و "دور مخصص"
- ❌ لا يوجد permissions JSON — الصلاحيات مش encode-able

2️⃣ جدول user_roles (Join table)

CREATE TABLE user_roles (
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  role_id UUID NOT NULL REFERENCES roles(id) ON DELETE CASCADE,
  assigned_at TIMESTAMP NOT NULL,
  PRIMARY KEY (user_id, role_id)
);

ملاحظات:
- ✅ Composite PK — مستخدم + دور = فريد
- ❌ لا يوجد assigned_by — مَن عيّن الدور؟ (audit gap)
- ❌ لا يوجد expires_at — أدوار دائمة فقط

🎫 JWT Structure (من JwtTokenService.GenerateAccessToken)

Claims المُضمّنة:

new(JwtRegisteredClaimNames.Sub, user.Id.ToString()),       // user ID
new(JwtRegisteredClaimNames.Email, user.Email),
new(JwtRegisteredClaimNames.Jti, Guid.NewGuid().ToString()), // unique token ID
new("tenant_id", user.TenantId.ToString()),                  // ⚠️ custom claim
new("full_name", user.FullName),
// ثم لكل role:
new(ClaimTypes.Role, role)  // ← كل role كـ claim منفصل

Token Validation (Program.cs):

options.TokenValidationParameters = new TokenValidationParameters
{
    ValidateIssuer = true,
    ValidateAudience = true,
    ValidateLifetime = true,
    ValidateIssuerSigningKey = true,
    ValidIssuer = jwtSettings.Issuer,
    ValidAudience = jwtSettings.Audience,
    IssuerSigningKey = new SymmetricSecurityKey(Encoding.UTF8.GetBytes(jwtSettings.Secret)),
    ClockSkew = TimeSpan.FromSeconds(30),
    NameClaimType = JwtRegisteredClaimNames.Sub,
    RoleClaimType = ClaimTypes.Role,  // ← مهم للـ [Authorize(Roles=...)]

TenantMiddleware (Shared/MultiTenancy/TenantMiddleware.cs)

public async Task InvokeAsync(HttpContext context, ITenantContext tenantContext)
{
    var path = context.Request.Path.Value ?? string.Empty;
    var isPublic = IsPublicPath(path);

    if (context.User?.Identity?.IsAuthenticated == true)
    {
        var tenantClaim = context.User.FindFirst("tenant_id")?.Value;
        var userClaim = context.User.FindFirst(ClaimTypes.NameIdentifier)?.Value
                        ?? context.User.FindFirst("sub")?.Value;

        if (Guid.TryParse(tenantClaim, out var tenantId) &&
            Guid.TryParse(userClaim, out var userId))
        {
            tenantContext.Set(tenantId, userId);
        }
        else if (!isPublic)
        {
            _logger.LogWarning("Authenticated request without tenant_id claim on path {Path}", path);
        }
    }

    await _next(context);
    tenantContext.Clear();
}

ما يفعله:
- ✅ يسحب tenant_id + user_id من JWT
- ✅ يحقنها في ITenantContext (Scoped service)

ما لا يفعله:
- ❌ لا يتحقق أن tenant_id في الـ token = tenant_id في الـ URL/body
- ❌ لا يمنع user من tenant A من الوصول لـ tenant B (إذا توصل لـ token من tenant B)
- ❌ لا يتحقق أن tenant_id في الـ JWT موجود فعلاً في tenants table

الـ isolation الفعلي يعتمد على:
- مطوّر يضيف WHERE tenant_id = @tenantId في كل query
- لا يوجد Global Query Filter (مثل EF Core)

🚨 الثغرات الحرجة (Critical Gaps)

🔴 Gap #1: لا يوجد تمييز بين القراءة والكتابة (Read vs Write)

المشكلة: حتى Viewer (الموثّق "قراءة فقط") يستطيع إنشاء/حذف/تعديل.

الإصلاح المقترح:

// حالياً
[HttpGet]
public async Task<IActionResult> List(...) { ... }

[HttpPost]
public async Task<IActionResult> Create(...) { ... }

// المقترح
[HttpGet]
[Authorize]  // أي مسجّل
public async Task<IActionResult> List(...) { ... }

[HttpPost]
[Authorize(Roles = "Admin,Accountant")]  // فقط Admin + Accountant
public async Task<IActionResult> Create(...) { ... }

🔴 Gap #2: Manual Retry مفتوح للجميع

المشكلة: POST /api/events/retry/{id} موثّق "Admin: manual retry" في الـ comment، لكن لا يوجد [Authorize(Roles = "Admin")]. أي محاسب يمكنه إعادة محاولة حدث يدوياً.

التأثير: لو المحاسب يعيد محاولة حدث فيه خطأ محاسبي، قد يُنشأ قيد مكرر.

الإصلاح:

[HttpPost("retry/{id:guid}")]
[Authorize(Roles = "Admin")]  // ← الإصلاح
public async Task<IActionResult> Retry(Guid id, CancellationToken ct) { ... }

🔴 Gap #3: Posting Rule Create/Trigger مفتوح للجميع

المشكلة: POST /api/finance/posting-rules — أي مسجّل يقدر ينشئ قاعدة (تأثير على القيود التلقائية).

التأثير: مستخدم خبيث يضيف قاعدة تنشئ قيود مكررة أو في حسابات خاطئة.

الإصلاح:

[HttpPost]
[Authorize(Roles = "Admin,Accountant")]
public async Task<IActionResult> Create(...) { ... }

[HttpPost("trigger/{eventType}")]
[Authorize(Roles = "Admin,Accountant")]
public async Task<IActionResult> Trigger(...) { ... }

🟠 Gap #4: لا يوجد Audit Log لعمليات Role

المشكلة: UserRole لا يحفظ assigned_by أو assigned_reason. مَن عيّن Admin لـ X؟ متى؟

الإصلاح المقترح:

ALTER TABLE user_roles ADD COLUMN assigned_by UUID REFERENCES users(id);
ALTER TABLE user_roles ADD COLUMN assigned_reason VARCHAR(200);
ALTER TABLE user_roles ADD COLUMN revoked_at TIMESTAMP;
ALTER TABLE user_roles ADD COLUMN revoked_by UUID REFERENCES users(id);

🟠 Gap #5: لا يوجد Soft Revoke للأدوار

المشكلة: user_roles join table — لو حذفنا row، نخسر التاريخ. ما نعرف كان عند المستخدم دور قبل.

الإصلاح: إضافة revoked_at (soft delete) + filter في queries.

🟠 Gap #6: لا يوجد Concept of "Permissions"

المشكلة: الأدوار = strings ("Admin", "Accountant"). لا يوجد جدول permissions منفصل.

مثال على ما نريد:

Role "Accountant" {
  permissions: [
    "journal_entries.create",
    "journal_entries.post",
    "accounts.read",
    "ledger.read",
    "reports.finance.read"
  ]
}
Role "Viewer" {
  permissions: [
    "*.read"  // wildcard
  ]
}

الإصلاح المقترح (Phase 4):
- جدول permissions (id, name, description)
- جدول role_permissions (role_id, permission_id)
- IRequirePermission attribute
- IAuthorizationHandler مخصص

🟠 Gap #7: لا يوجد Tenant-Aware Role Check

المشكلة: عند Admin من tenant A يحاول يدخل tenant B — الكود لا يمنع ذلك (إذا وصل لـ token من tenant B).

الإصلاح:

// في TenantMiddleware
if (Guid.TryParse(tenantClaim, out var tenantId))
{
    var tenantExists = await _tenants.ExistsAsync(tenantId, ct);
    if (!tenantExists)
    {
        context.Response.StatusCode = 403;
        return;
    }
}
