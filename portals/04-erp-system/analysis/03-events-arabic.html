🔔 الأحداث والربط بين الموديلات (Events & Cross-Module) — شرح بالعربي

> التاريخ: 2026-07-21

> المؤلف: المختبر التحليلي (Mavis Lab)

> الجمهور: الفريق غير التقني (أنس + صاحبه)

> المصدر: live DB via Neon MCP

🎯 شو هي "الأحداث"؟

الأحداث هي الرسايل اللي تطلع من موديل لما يصير شي مهم، والموديلات الثانية تسمعها وتتفاعل معها.

مثال واقعي: 

في الـ ERP:
1. المبيعات يطلع فاتورة → حدث "invoice_created"
2. Finance يسمع → يعمل قيد يومية تلقائي
3. المخزون يسمع → ينقص الكمية
4. الإشعارات تسمع → تبعت notification للعميل
5. التقارير تسمع → تحدد الإحصائيات
بدون الأحداث: الموديلات تكون "جزر منعزلة" — كل واحد يشتغل لحاله. مع الأحداث: النظام كله مترابط و ذكي.

📬 طاولة outbox_events (صندوق الصادر)

شو هي؟

كل حدث في النظام يسجل هنا — قبل ما يتنفذ. زي "صندوق بريد الصادر" في الدوائر الحكومية.

مثال على event:


{
  "id": "uuid-1234",
  "tenant_id": "alfajr-trading",
  "event_type": "sales_invoice.created",
  "aggregate_id": "INV-2026-0042",
  "aggregate_type": "sales_invoice",
  "payload": "{...تفاصيل الفاتورة...}",
  "occurred_at": "2026-07-21T15:30:00Z",
  "processed_at": null,
  "retry_count": 0,
  "max_retries": 5,
  "last_error": null
}

الحقول:

الحقل المعنى
`event_type` نوع الحدث (مثلاً `sales_invoice.created`)
`aggregate_id` ID الكيان المرتبط (رقم الفاتورة مثلاً)
`aggregate_type` نوع الكيان (`sales_invoice`، `payment`، ...)
`payload` محتوى الحدث (JSON بكل التفاصيل)
`occurred_at` متى وقع الحدث
`processed_at` متى اتعالج (null = لسه ما اتعالج)
`retry_count` كم محاولة فشلت
`max_retries` الحد الأقصى للمحاولات
`last_error` رسالة الخطأ الأخيرة (لو فشلت)

✅ طاولة processed_events (الأحداث المعالجة)

شو هي؟

سجل بكل الأحداث اللي تمت معالجتها بنجاح. مهم للـ idempotency (ضمان عدم المعالجة المزدوجة).

مثال:


{
  "event_id": "uuid-1234",
  "tenant_id": "alfajr-trading",
  "processed_at": "2026-07-21T15:30:05Z"
}

ليش مهمة؟

بدون processed_events: لو الـ event اتبعت مرتين (network issue) → قيد يتسجل مرتين → خطأ محاسبي! مع processed_events: قبل ما يعالج event، يتحقق لو event_id موجود في processed_events:
  • ✅ موجود → skip (عُولج قبل)
  • ❌ مش موجود → عالج + سجّل في processed_events
    • هذا يضمن:
  • مرة واحدة فقط لكل event
  • حتى لو فشلت المعالجة، retry ما يكرر
  • Audit trail نظيف

    • 📨 طاولة notifications (الإشعارات)

    شو هي؟

    الإشعارات اللي تطلع للمستخدمين (in-app). زي inbox صغير.

    مثال:

    
{
  "id": "uuid-9999",
  "tenant_id": "alfajr-trading",
  "user_id": "user-anas",
  "type": "project_budget_exceeded",
  "title": "تجاوز ميزانية المشروع",
  "message": "مشروع P-2026-001 صرف 95% من الميزانية",
  "reference_type": "project",
  "reference_id": "uuid-project-1",
  "is_read": false,
  "created_at": "2026-07-21T15:30:00Z",
  "read_at": null
}

    الحقول:

    الحقل المعنى
    `user_id` مين يستلم الإشعار
    `type` نوع الإشعار (مفتاح للـ icon واللون)
    `title` العنوان القصير
    `message` النص الكامل
    `reference_type` نوع الكيان المرتبط (project, invoice, ...)
    `reference_id` ID الكيان (للضغط عليه يفتح)
    `is_read` هل المستخدم قرأه؟
    `read_at` متى قرأه

    أنواع الإشعارات الشائعة:

    
# Budget alerts
"project_budget_warning"  # 80% من الميزانية
"project_budget_exceeded"  # 100%+
"project_budget_critical"  # 110%+

# Approval requests
"po_pending_approval"
"invoice_pending_approval"
"leave_request_pending"

# Status changes
"po_approved"
"po_rejected"
"invoice_paid"
"payment_received"

# System alerts
"user_login_new_device"
"password_changed"
"role_assigned"

    📝 طاولة audit_log (سجل التدقيق)

    شو هي؟

    سجل كامل بكل التغييرات اللي صارت على البيانات — مين عمل شنو ومتى.

    مثال:

    
{
  "id": 12345,
  "tenant_id": "alfajr-trading",
  "entity_type": "sales_invoice",
  "entity_id": "INV-2026-0042",
  "action": "updated",
  "user_id": "user-anas",
  "changes": {
    "before": {"status": "draft", "total": 5000},
    "after": {"status": "posted", "total": 5000}
  },
  "ip_address": "192.168.1.10",
  "created_at": "2026-07-21T15:30:00Z"
}

    الحقول:

    الحقل المعنى
    `entity_type` نوع الكيان (`sales_invoice`, `project`, ...)
    `entity_id` ID الكيان
    `action` الإجراء (`created`, `updated`, `deleted`, `viewed`, `exported`)
    `user_id` مين عمل الإجراء
    `changes` JSON فيه قبل وبعد التغيير
    `ip_address` IP المستخدم
    `created_at` متى

    ليش مهمة؟

  • 🔍 **تتبع الأخطاء** — "مين عدّل هذا الرقم؟"
  • 🛡️ **الأمان** — كشف التلاعب
  • 📋 **Compliance** — مطلوب قانونياً في كثير من الدول
  • 👨‍⚖️ **حل النزاعات** — "الفاتورة كانت X أو Y؟"

    • 🔄 كيف يشتغل النظام كاملاً (Flow كامل)

    مثال واقعي: "فاتورة مبيعات جديدة"

    
الخطوة 1: المستخدم يعمل فاتورة
  ↓
  sales_invoice table: row جديد
  ↓
الخطوة 2: النظام يطلع events
  ↓
  outbox_events: 
    - event_type: "sales_invoice.created"
    - payload: {invoice_id, customer, total, ...}
  
الخطوة 3: مستمعين الأحداث يسمعون
  ↓
  ┌─────────────────────────────────────┐
  │  Finance listener:                  │
  │    - استلم event                    │
  │    - طبق posting_rule               │
  │    - أنشأ journal_entry + lines     │
  │    - سجّل في processed_events       │
  └─────────────────────────────────────┘
  ┌─────────────────────────────────────┐
  │  Inventory listener:                │
  │    - استلم event                    │
  │    - أنقص الكمية من stock_levels    │
  │    - سجّل stock_movement            │
  │    - سجّل في processed_events       │
  └─────────────────────────────────────┘
  ┌─────────────────────────────────────┐
  │  Notifications listener:            │
  │    - أنشأ notification للعميل        │
  │    - سجّل في processed_events       │
  └─────────────────────────────────────┘
  ┌─────────────────────────────────────┐
  │  Audit listener:                    │
  │    - سجّل في audit_log              │
  │    - سجّل في processed_events       │
  └─────────────────────────────────────┘
  
الخطوة 4: outbox_events → processed_at = now()
الخطوة 5: sales_invoice: outbox marked as processed
    كل هذا في ثوانٍ معدودة — أوتوماتيكي بالكامل.

    🛡️ ضمانات الأمان (Reliability)

    1. Idempotency (مرة واحدة فقط)

  • `processed_events` يمنع المعالجة المكررة
  • حتى لو الـ event اتكرر (network glitch)

    • 2. Retry Logic (إعادة المحاولة)

  • لو فشلت المعالجة → `retry_count++`
  • لحد `max_retries` (عادة 5)
  • لو وصلت الحد → `last_error` يُحفظ + alert

    • 3. Transactional Outbox Pattern

  • الـ event يحفظ **في نفس الـ transaction** مع الـ data
  • لو الـ data ما اتحفظت → الـ event ما يتحفظ
  • ضمان: مفيش event بدون data، مفيش data بدون event

    • 4. Eventual Consistency

  • الـ events ممكن تتأخر ثواني/دقائق
  • لكن ما تضيع أبداً
  • eventually consistent state

    • 🎯 ملخص سريع

    الطاولة الدور
    `outbox_events` صندوق الصادر (كل الأحداث)
    `processed_events` اللي اتعالج (لـ idempotency)
    `notifications` إشعارات المستخدمين (in-app)
    `audit_log` سجل التدقيق (مين عمل شنو)
    النمط: Transactional Outbox + Idempotency الفايدة: الموديلات مترابطة + آمنة + قابلة للتوسع

    • 🧠 الموديلات الثانية تستمع لـ events من Finance و Projects:

    Finance يبعت events عند:

  • ✅ قيد مُرحّل → `journal_entry.posted`
  • ✅ دفعة مستلمة → `payment.received`
  • ✅ تجاوز ميزانية → `budget.exceeded`

    • Projects يبعت events عند:

  • ✅ مشروع جديد → `project.created`
  • ✅ تغيير حالة → `project.status_changed`
  • ✅ تجاوز 80% من الميزانية → `project_budget_warning`
  • ✅ تجاوز 100% → `project_budget_exceeded`
  • ✅ مهمة مكتملة → `project_task.completed`
  • ✅ مشروع مكتمل → `project.completed`

    • اللي يسمع:

  • **HR**: يحسب تكلفة المشروع من ساعات العمل
  • **Procurement**: يربط POs بالمشروع
  • **Inventory**: ينقص الكمية
  • **Reports**: يحدّث KPIs
  • **Notifications**: يبعت للمستخدمين
    • النتيجة: نظام واحد متماسك، كل موديل يعرف شو يصير في الباقي بدون coupling مباشر.
