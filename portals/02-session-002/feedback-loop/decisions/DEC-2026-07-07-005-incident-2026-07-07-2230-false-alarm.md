## 🔍 Root Cause

### HF Spaces Case-Sensitive Hostname Behavior:

When you create an HF Space with a user/space URL like `Anas-Assaket/erp-system`, HF generates a subdomain based on the user-defined mixed-case form. The HF **proxy** (front-end) routes requests case-sensitively:

| URL Form | Behavior |
|---|---|
| `https://anas-assasket-erp-system.hf.space` (lowercase) | Routed to a 404 path → HF's standard "Sorry, we can't find the page" HTML |
| `https://Anas-Assaket-erp-system.hf.space` (mixed case) | Routed correctly to the Docker container → 200 + actual API |

This is **HF-specific behavior** — most proxies are case-insensitive but HF opted for explicit match.

### Why Both URLs Resolve via DNS:

DNS `getent hosts` returns the same IPs for both forms (35.175.32.85, 3.223.204.224, 52.72.113.251). So at the **DNS layer** both URLs work. The 404 comes from **HF's HTTP proxy layer** post-DNS-resolution.

### Why Mavis Was Correct:

Mavis's curl command (and earlier probes in this session) used the canonical URL via:
```
https://anas-assaket-erp-system.hf.space/api/...
```
But his terminal/environment may have normalized to mixed case, OR he used the hostname dynamically from the HF API response.

## 🛡️ Defense Layer 34 — HTTP-Probe-With-Canonical-URL

### Proposed Rule (cross-project):

Any HTTP probe against an HF Space MUST use the **canonical mixed-case URL** as returned by HF's `/api/spaces/<owner>/<space>` endpoint, never a lowercased or modified form.

### Implementation Pattern:

```bash
# Always fetch the canonical hostname dynamically
HF_RESP=$(curl -s "https://huggingface.co/api/spaces/<owner>/<space>")
CANONICAL_HOST=$(echo "$HF_RESP" | python3 -c "import json,sys; print(json.load(sys.stdin).get('host',''))")

# Use EXACTLY as HF returns (preserves case)
code=$(curl -s -o /dev/null -w "%{http_code}" --max-time 8 \
  "${CANONICAL_HOST}/api/health/live")

if [ "$code" != "200" ]; then
  echo "🚨 Health check failed: HTTP $code"
  exit 1
fi
```

### Why This Matters:

- HF stage check (`Stage: RUNNING`) is **not** a reliable indicator that the app is actually responding
- Lowercase URL probe gives false positives (false 404 → false alert)
- Canonical URL probe is the only reliable way to check if the app is serving
